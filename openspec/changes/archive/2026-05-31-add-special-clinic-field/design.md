## 背景

目前醫院資料使用 `type`、`typeText`、`pet_category_group`、`services`、`specialties` 與自由文字 `clinicNotes` 來描述特寵診療能力。這足以表示「此醫院支援特寵」，但不足以區分「一般時段皆可看特寵」與「只在特別門診、固定每週時段、指定醫師或特殊預約流程中看特寵」。

## 目標 / 非目標

**目標：**
- 用輕量 optional 欄位表示特寵特別門診。
- 保持既有醫院分類與篩選行為穩定。
- 在列表與詳情頁提示特別門診限制，且不要求所有醫院資料一次補齊。
- 保留來源與確認日期，方便後續維護與查核。

**非目標：**
- 不在這次建立完整的每週特別門診排班、醫師、物種對應或預約矩陣。
- 不改動 `type`、`typeText` 或目前「非預約制」篩選行為。
- 不在這次一次遷移所有疑似特別門診的既有備註。

## 設計決策

- 新增 optional 的 `Hospital.specialClinic` object，而不是新增 `type` 值。這能維持地圖與分類行為穩定，並把特別門診視為「可用性限制」而不是新的醫院類型。
- 使用 `hasExoticSpecialClinic` 作為畫面顯示觸發條件。維護者可以先補來源資訊，但只有在確認屬於特別門診或限定特寵門診時才顯示標籤。
- `note` 維持簡短人工可讀文字，而不是結構化排班資料。現有來源品質不一，且目前 UI 只需要可靠提醒，不需要做到日曆級可用性判斷。
- `specialClinic.reservationRequired` 與 `Hospital.reservationRequired` 分開。部分醫院全院可能可現場掛號或混合掛號，但特寵特別門診本身仍可能需要預約。
- `services`、`specialties` 與 `clinicNotes` 繼續作為服務標籤與較完整備註。新欄位是補充，不取代既有文字欄位。

## 風險 / 取捨

- 特別門診時段仍是自由文字 -> 以 `note`、來源與確認日期降低誤解風險。
- 部分既有資料可能仍只在 `clinicNotes` 隱藏特別門診限制 -> 之後在資料複查時逐步遷移。
- 列表新增標籤可能讓卡片更密 -> 使用短標籤，且只有 `hasExoticSpecialClinic` 為 true 時才顯示。
- 維護者可能混淆兩個預約欄位 -> 文件明確規定 `specialClinic.reservationRequired` 只適用特別門診，不影響既有「非預約制」篩選。
