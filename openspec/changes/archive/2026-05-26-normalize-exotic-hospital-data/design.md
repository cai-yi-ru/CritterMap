## Context

CritterMap 現有 16 個縣市的醫院資料檔案，合計約數百筆特寵醫院資料（`type: "exotic"` 或 `typeText` 含「特寵」）。這些資料由不同人員建立，欄位完整度不一：

- `pet_category_group` 有時為空陣列，導致寵物種類篩選失效
- `business_hours` 格式有時為字串而非時段陣列
- `typeText` 表達多樣（「特殊寵物診療」、「犬貓與特寵診療」、「特寵」等）
- `reservationRequired`、`hasEmergencyService`、`nightClinic` 等布林欄位常為 `undefined`
- `last_checked` 欄位普遍缺漏

## Goals / Non-Goals

**Goals:**
- 建立一套特寵醫院資料填寫規範（spec）
- 逐一審查並補全 16 個縣市的特寵醫院欄位
- 統一 `typeText` 用語
- 確保 `pet_category_group` 與 `pets` 欄位對應正確
- 補充 `last_checked` 日期

**Non-Goals:**
- 不新增醫院資料（僅整理現有資料）
- 不修改 `Hospital` 型別定義（src/types/hospital.ts）
- 不修改前端篩選邏輯
- 不驗證醫院地址或電話的真實性

## Decisions

### 決策 1：typeText 標準化用語

採用以下兩種標準 typeText：
- **`"特殊寵物診療"`**：醫院主要或專門診療特殊寵物（兔、鼠、刺蝟、爬蟲等），不以犬貓為主業
- **`"犬貓與特寵診療"`**：醫院以犬貓為主，兼具特寵診療能力

棄用其他表達（如「特寵」、「特殊寵物」、「特殊寵物看診」）。

### 決策 2：pet_category_group 標準分類

採用現有系統定義的分類集合：
```
["狗", "貓", "兔", "鼠", "天竺鼠", "鳥類", "爬蟲", "刺蝟", "蜜袋鼯", "貂", "龍貓", "兩棲", "野生動物", "其他特寵"]
```
根據 `pets` 欄位內容對應填入 `pet_category_group`，對應規則：
- 倉鼠、黃金鼠、三線鼠 → `"鼠"`
- 狗、犬 → `"狗"`
- 龍貓 → `"龍貓"`
- 貂、雪貂 → `"貂"`
- 兩棲類 → `"兩棲"`
- 野生動物 → `"野生動物"`
- 蜜袋鼯 → `"蜜袋鼯"`
- 各種爬蟲（蜥蜴、蛇、烏龜等）→ `"爬蟲"`

### 決策 3：逐檔處理策略

每個縣市檔案獨立處理，避免大規模批次修改造成混亂。處理順序依城市規模由大到小（台北 → 新北 → 高雄 ... → 台東）。

### 決策 4：last_checked 日期

補充缺漏的 `last_checked` 欄位，值填入本次整理日期（`"2026-03-25"`）。

## Risks / Trade-offs

- **[風險] 資料判斷不確定性** → 無法即時聯繫醫院確認資料，以現有 `hours`、`services` 文字為依據推斷，標注需人工複核的項目
- **[風險] 修改量大造成 review 困難** → 以縣市為單位分開 commit，便於追蹤與 rollback
- **[Trade-off] 不查詢外部資料** → 資料補全依賴現有欄位推斷，準確度有限，但確保不引入未驗證資訊
